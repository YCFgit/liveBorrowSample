## 一、项目背景与目标

### 1.1 一期解决什么问题

| 痛点 | 现状 |
| --- | --- |
| **入口分散** | 截图、口头沟通、邮件发起，无统一入口 |
| **寻源靠人工** | 货控手工匹配门店库存，效率低 |
| **物流单号缺失** | 门店人工呼叫快递后常不填单号，无法追踪 |
| **无全流程追溯** | 借出和归还是两张独立调拨单，无法关联 |
| **品相不可控** | 借出时无记录，归还时线下沟通，争议无据 |

### 1.2 一期目标

*   **钉钉统一入口**：卡片式提交，自动校验
    
*   **自动寻源派单**：助手自动寻源后调用GMS接口自动生成调拨单
    
*   **全流程可追溯**：借样单号串联借出和归还
    
*   **物流事后补录**：支持钉钉端补填，调用GMS回填接口
    
*   **质检销账闭环**：通过唯一码管控流程B/C品留痕
    
*   **自提全流程支持**（V1.1新增）：借样人自助发起自提、自助核销提货、自助确认归还（到店还样）
    
*   **通用批量归还**（V1.2新增）：支持按虚店聚合商品、先进先出自动匹配、自提样品快递/自选方式归还

## 三、核心业务节点

### 阶段一：发起与发货

**流程现状**

1.  借样发起：借样方通过邮件把需求发给货控，货控凭人工判断后，然后手工在GMS里新建调拨单。
    
2.  门店发货：收到调拨单后拣货，人工叫快递发出。近一周涉及借样店收货调拨单中56%为门店自提，其中25%明确备注为真自提，其余75%实际为快递发货，未记录快递单。
    

**改造流程**

1.  借样发起：直接通过钉钉@借样助手，助手推送卡片内容点击填写借样申请，并自动完成审核
    
    1.  审核规则：
        
    
    | 审核项 | 规则项 | 内容 |
    | --- | --- | --- |
    | 借样人审核 | 是否允许借样 | 判断工号下是否维护有借样虚店 |
    |  | 是否有借样逾期 | 判断工号下是否有逾期15天+未还借样单 |
    |  | 是否已达借样上线 | 判断工号+虚店下是否借样中总量超特定值（暂定200件）依据直播间规模制定 |
    | 借样商品审核 | 是否黑名单商品 | 黑名单商品标识（QS、上市月等） |
    |  | 借样库存是否充足 | 是否存在可用库存/降级±1个尺码是否有库存 |
    
2.  自动寻源：审核后生成借样单号，并基于借样申请内容，查询现有可用库存项匹配可用发货地址
    
    1.  寻源优先级：
        
        1.  发货地物流成本（考虑拆合单成本）
            
        2.  B品库存门店--生成调拨单后，备注优先出B品
        
    2.  GMS建单：
        
        1.  基于收发地址+商品明细生成类型为样品调货的调拨单，确认人为借样助手
    
3.  门店发货：直接通过现有流程完成发货，返回更新发货状态
    
    1.  现有流程自动代入快递收货人，可直接生成物流方式为快递的调拨单--可直接联动系统呼叫快递单完成快递流程闭环（现阶段除丽讯物流外均维护为自提并人工呼叫快递无法获得单号）
        
    2.  基于关联的快递单号，可通过丽讯接口获得快递签收状态
        

**新增：自提借样发起（V1.1）**

• 在借样申请卡片中增加 物流方式 单选框：快递 / 自提，默认快递。

• 若选择 自提，则展示 自提门店输入框，借样人必须填写一个门店（输入具体门店编号，不允许模糊查找）。

• 提交时校验：

◦ 物流方式不能为空

◦ 若为自提，必须选择有效门店

• 提交成功后，系统生成自提类型的借样任务，初始状态为 待自提

**上下游数据对接**

1.  **GMS新建调拨单流程：审核后生成借样单据--现有能力**
    
2.  **GMS可用库存查询：基于sku+siza获得可派备选库存--现有能力**
    
3.  **丽讯物流状态更新：基于物流单号查询快递签收时间--现有能力，目前审计在用**
    
4.  **GMS调拨单查询：通过借样单号关联的GMS调拨单号，查询调拨单状态与快递单号，以更新发货状态**
    
5.  **GMS回写物流单号：人工维护物流后更新单据快递--现有能力**
    
6.  **GMS出库唯一码管控：现阶段仅支持非A品出库，须改造，流程在GMS内闭环不涉及对接**
    

### 阶段二：收货与使用

**流程现状**

借样方收到货后，线下告知对接人“收到了”，对接人在GMS里手工操作调拨单收货，流程完结。

**改造流程**

1.  自助收货：借样任务发货中生成发货通知，通知颗粒度为借样单号+快递单号；针对快递单号级别可进行收货确认；确认后自动更新GMS调拨单收货状态
    
2.  差异反馈：收货流程存在问题反馈留痕
    

#### 新增：自提核销（自助确认已提货）（V1.1）

*   借样人实际到店拿到商品后，在任务详情页点击 **“确认已自提”**。
    
*   页面展示商品清单，支持：
    
    *   勾选本次实际拿到的商品（可部分勾选）
        
    *   修改本次实际拿到的数量（≤ 该商品借样数量）
    
*   提交后：
    
    *   更新商品的 `已提数量`
        
    *   若全部商品已提 → 任务状态变为 `借样中`
        
    *   若部分商品已提 → 任务状态保持 `待自提`，剩余商品可再次确认
    
*   **由提货方直接确认收货**。（是否需要门店复核？）
    
*   **差异反馈：自提场景下，借样人若发现商品数量/规格不符在线下自行沟通处理，暂不支持反馈流程**
    

**上下游数据对接**

1.  **GMS调拨单数据更新：基于GMS调拨单号更新调拨单状态为完结**
    
2.  **调拨异常差异反馈-钉钉：提醒流程如何闭环待沟通（货品负责人处理，通过钉钉跟进，具体负责人找人逻辑待细化）**
    

### 阶段三：归还与质检

**现有流程**

1.  借样归还：借样方线下和货控沟通，货控手工做一张反向调拨单。借样方呼叫快递寄出或走丽讯物流，若为快递则无单号跟进。
    
2.  仓库质检：调拨单寄回后由收件方确认收货完结，由于寄回快递无单号，可能变为无头件拆包，最终无法追溯品相情况
    

**改造流程**

**任务单归还**

1.  借样归还：通过助手卡片自助发起还样流程，自助生成还样调拨单
    
2.  归还寻仓：依据归还寻仓规则提供还样地址
    
    1.  整单归还可流转至门店或其他虚店（规则待沟通）
        
    2.  **整单流转默认回仓**
    
3.  物流录入：绑定快递物流单号，关联还样调拨单
    
4.  仓库质检：还样调拨单绑定的物流单号收货自动关联收件--后续唯一码流程覆盖后可直接通过唯一码流水识别样品来源；
    

####  新增：自提归还（自助确认归还 ）（V1.1）

 **借样人自助确认归还**

*   自提任务在 `借样中` 状态时，任务详情页显示 **“确认已归还（自提）”** 按钮。
    
*   点击后展开商品清单，支持：
    
    *   勾选本次实际归还的商品（可部分勾选）
        
    *   修改本次归还数量（≤ 该商品已提数量）
    
*   提交后：
    
    *   更新商品的 `已归还数量`
        
    *   任务状态变为 `待门店收货`
        
    *   若部分归还，剩余商品仍可再次发起归还
        

**新增：通用归还功能集成 （V1.2）**

##### 通用归还入口

*   在现有 **“发起借样”** 按钮旁边，增加 **“📦 发起还样”** 按钮。
    
*   点击后跳转到 **通用还样页面**，作为所有归还操作的统一入口。
    
*   原有单个任务详情页的“发起归还”按钮 **保留不变**，两者可并存。
    

#####  通用还样页面设计

 **选择虚店**

*   下拉选择框，列出用户有权限的所有虚店。
    
*   选择后自动加载该虚店下的商品清单。
    

3.   **样品类型筛选（两个 Tab）**


*   **全部样品**（默认）：展示所有样品（包括快递样品和自提样品）。
    
*   **仅自提样品**：只展示自提样品。
    

1.   **商品清单（汇总视图）**


展示字段：

*   复选框
    
*   商品编码 + 尺码
    
*   可还总数量（该虚店下所有借样中任务中，该 SKU+尺码的剩余未还数量之和）
    
*   本次归还数量输入框（数字，默认0，不可超过可还数量）
    
*   **若为自提样品**，额外展示小字：`来源门店：XXX`（取任一关联任务的 storeName）
    

数据来源：后端实时聚合计算（按虚店 + 商品 + 尺码，汇总可还数量，并记录对应的任务列表）。

 **归还方式选择（动态显示）**

*   **仅在当前筛选为“仅自提样品”时**，在商品清单下方、汇总栏上方显示归还方式选择器。
    
*   选项：`📦 快递归还`（默认）、`🏪 自行归还（到店）`。
    
*   当筛选为 **“全部样品”** 时，**不显示**归还方式选择器，系统默认所有勾选的样品（无论类型）均走 **快递归还**。
    
*   **汇总栏**
    
*   展示：已选择商品种类数、合计归还件数，实时更新。
    

 **提交按钮**

*   文案：`确认还样（X件）`
    
*   点击后触发后端分配逻辑。
    

#####  先进先出自动分配

后端根据用户勾选的商品及数量，按以下规则分配到具体借样任务：

*   按任务借出日期升序排序（最早的先还）。
    
*   同一商品可能拆到多个任务中，直到满足用户填写的归还数量。
    
*   生成 **归还批次号**（格式：`RT` + yyyyMMdd + 6位随机数，如 `RT20260601000001`），并写入每个被分配的任务单。
    

#####  根据筛选页面和用户选择分类处理

| 用户场景 | 系统行为 |
| --- | --- |
| **在“全部样品”页面提交**（默认快递） | 所有勾选商品（含自提）统一使用 **快递归还**。每个关联任务的 `returnMethod = 'express'`，`returnStatus = 'pending'`（待填写物流）。 |
| **在“仅自提样品”页面提交，并选择“快递归还”** | 每个关联任务的 `returnMethod = 'express'`，`returnStatus = 'pending'`，需填写物流单号。 |
| **在“仅自提样品”页面提交，并选择“自行归还（到店）”** | 每个关联任务的 `returnMethod = 'in_person'`，`returnStatus = 'store_pending'`（待门店收货），无需物流单号。 |

> 注意：一次提交中的所有自提样品只能选择同一种归还方式（由用户在归还方式选择器中统一选择），暂不支持混合。

#####  提交结果页

*   展示 **归还批次号**。
    
*   列出本次关联的所有借样任务（任务号、商品明细、归还数量、归还方式）。
    
*   若为快递归还，提供 **\[填写物流单号\]** 按钮，点击后跳转到物流录入页（单号写入该批次号下所有任务）。
    
*   若为自行归还，提供 **\[查看详情\]** 按钮，可跳转到各任务详情页查看状态。
    
*   提供 **\[返回任务列表\]** 按钮。
    

#####  物流录入（快递场景）

*   用户填写快递公司、快递单号。
    
*   提交后，系统将单号写入该批次号下所有 `returnMethod = 'express'` 且 `returnStatus = 'pending'` 的任务的 `returnLogisticsNo` 字段，并更新 `returnStatus = 'logistics_filled'`。
    
*   后续仓库收货、质检流程与原有保持一致（调用GMS回写物流单号、调拨单查询等）。
    

#####  自行归还后续流程（复用V1.1自提归还设计）

*   用户提交自行归还后，任务状态变为 `store_pending`（待门店收货）。
    
*   在任务详情页提供 **“模拟门店收货并质检通过”** 的演示入口（或正式对接门店POS）。
    
*   门店确认收货后，任务进入质检流程，最终销账。
    

##### 数据模型扩展（新增字段）

在借样任务单（task）中增加以下字段：

| 字段名 | 类型 | 说明 |
| --- | --- | --- |
| `returnBatchNo` | string | 归还批次号，为空表示未发起归还。一次归还操作生成唯一批次号，多个任务可共享同一批次号 |
| `returnStatus` | enum | 本任务在本次归还中的状态：`pending`（已发起）、`logistics_filled`（物流已填，仅快递）、`store_pending`（待门店收货，仅自行）、`quality_checking`（质检中）、`completed`（已销账） |
| `returnedQty` | int | 本任务中已发起归还的数量（支持部分归还） |
| `returnLogisticsNo` | string | 物流单号（快递场景），同一批次号下的多个任务可冗余存储相同单号 |
| `returnMethod` | enum | 归还方式：`express`（快递）、`in_person`（自行归还到店） |

#####  与现有流程的兼容性

| 现有功能 | 兼容方式 |
| --- | --- |
| 单个任务详情页的“发起归还”按钮 | 保留不变，点击后为该任务单独生成一个归还批次号（只关联自己），并允许用户选择归还方式（自提任务才显示选择） |
| 快递归还的物流单号录入 | 复用现有逻辑，但需支持同一批次号下批量更新 |
| 自提归还的模拟门店收货 | 保持原有演示入口，不冲突 |
| 唯一码7天品相监控 | 完全复用，按每个任务独立进行 |

2.  **还样物流呼叫（待讨论）：直播间可直接通过系统呼叫回程快递并关联**

### 阶段四：销账闭环

**流程现状**

以逆向调拨单为准，无系统跟进销账

**改造流程**

收方结构收货后，系统7天内轮巡唯一码品相：

**A品**：自动触发销账，任务单完结，借样方收到“销账完成”通知。整个借样闭环结束。

**B/C品**：分两种情况。如果归还品相和借出时一致（借出时就是B品，回来还是B品），任务直接完结，记录留痕。如果品相降级了（借出时是A品，回来变成B或C品），系统标记为“待处理”，留到三期做赔付流程。虚店里的库存通过“虚店销售”的方式核销，账实一致。

**上下游数据对接**

1.  **唯一码查询接口：基于借样单号中的唯一码信息，查询流水品相变化**
